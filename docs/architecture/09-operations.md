# 09 — Per-Operation Design

Ladders summarized in [`04`](04-capability-router-and-ladder.md); contracts in [`05`](05-driver-contracts.md). `[data]` → [`background/benchmark-summary.md`](background/benchmark-summary.md).

## Shipped drivers & operations (as-built status)

This table is the completeness ledger — every first-party driver and op, with its **registration status**. "✅ auto" = in [`../../src/drivers/defaults.ts`](../../src/drivers/defaults.ts), so the zero-config engine routes to it on a capability miss (ADR-004/006). Some auto entries are **lazy proxies**: their cheap `supports()` predicate is registered in the default bundle, while the heavy driver implementation is imported only when that route is actually selected (ADR-103). Production first-party container/filter/image drivers and the real Symphonia audio WASM tails are auto-registered; scaffold codec tails stay out of defaults until their cores exist. The still/animated **image capability** is auto-registered too, but it is not a `ContainerDriver`/`CodecDriver` because images have no packet seam (ADR-049). The **streaming-output** implementation is split deliberately: the CMAF writer and `StreamTarget` sink are wired into the public sink/remux/mux surface (ADR-034/046), URL-backed MP4 stream-copy coalesces source sample range reads (ADR-052), same-container stream targets use lazy progressive source-range copy unless fragmentation is explicitly requested (ADR-101), cross-container WebM/MKV remux can use a true Cluster-on-write streaming muxer for large or live targets (ADR-113), keyframe MP4/MOV trims can materialize only selected source ranges through the layout-only buffered path (ADR-114), and ISO-BMFF explicit buffer targets use bounded whole-buffer routes: ordinary outputs can use a single-allocation progressive source-range materializer, while GB-scale browser buffer rows may use fragmented whole-buffer output under the verified cap (ADR-102). The browser harness adapter now declares real `target:writes` telemetry from positioned callbacks and validated WebM `headerless` output (ADR-099/100), while formats without a bounded materializer still decline over-size buffer targets honestly instead of wedging (ADR-053).

**Container drivers** (`ContainerDriver`, hand-written TS). **Container-output** = whether the container can be a `convert`/`remux`/`encode` *target*, and by which mechanism — a chunk-seam `Muxer` (`createMuxer`, ADR-028/037), driver-native `streamCopy` (remux/trim, ADR-021), or the raw-PCM `transformPcm` path (ADR-022). **Codec drivers** (`CodecDriver`). The WebCodecs pair is the `tier:'hardware'` head; the `tier:'wasm'` tail is **miss-only** (built only when WebCodecs lacks the codec). The real Symphonia decode tails (`wasm-vorbis`, `wasm-aac`, `wasm-mp3`), the real `wasm-opus` encode+decode tail (vendored prebuilt permissive libopus-wasm, ADR-088), the real `wasm-av1` decode tail (vendored dav1d, ADR-093), the real `wasm-vpx` decode tail (ogv.js/libvpx, ADR-094), and the real `wasm-vorbis-enc` encode tail (libogg/libvorbisenc built with Emscripten, ADR-108) are auto-registered behind lazy proxies. `scripts/vendor-wasm.ts` co-vendors Rust-tail glue + `.wasm` assets next to `dist` (ADR-042) and skips self-contained inlined/Emscripten single-file cores where there is no separate `.wasm` to copy (ADR-090/108). Their `supports()` probes require the relevant browser host-frame seam (`EncodedAudioChunk` → `AudioData` or `EncodedVideoChunk` → `VideoFrame`) and now validate each core's exact codec envelope before reporting support (ADR-111); the heavy core is fetched/instantiated only when the coder stream starts (ADR-004/005/006). **Operations** (public verbs, doc 07): `probe` · `demux` · `remux` (mp4 stream-copy, ADR-021; MP4 edit-list packet timestamp normalization, ADR-071; packet-preserving FLAC→Ogg-FLAC, ADR-064; H.264/AAC→MPEG-TS with signed-preroll rebase, ADR-065/072; large WebM/MKV Cluster-on-write streaming remux, ADR-113) · `trim` (keyframe-aligned packet-copy, MP4/MOV lazy selected-range materialization, ADR-114, and accurate decode→frame-window→re-encode→mux seam, ADR-082) · `decrypt` (`cenc`/`cens`/`cbcs`/`hls-aes128`/`hls-sample-aes`, ADR-023/121) · `convert`/`transcode` (auto-routes PCM-native / compressed-to-WAV `decodePcm` ADR-022/024/050 → stream-copy ADR-021/068 → codec seam) · `decode` (lazy frame streams, ADR-030; still/animated images via `ImageDecoder`, ADR-049/077; raw PCM WAV/AIFF/CAF audio via `decodePcmAudio`, ADR-063; Vorbis/AAC/MP3 via auto-registered Symphonia wasm tails when the browser audio frame seam exists, ADR-036/039/069) · `encode` (ADR-029/030) · `mux` (chunk `Muxer` for MP4/WebM/Ogg/MP3/MPEG-TS, ADR-028/037/064/065/066/067/070/071/072/097/098; PCM via `transformPcm` for WAV/AIFF/CAF) · `seek` (frame-accurate, ADR-026; close-race-safe, ADR-040). **Tier split (ADR-025):** the pure-TS ops are Node-validated — probe/demux/remux/keyframe-trim/decrypt/PCM-convert/FLAC-decode and image probe, plus the MP4/WebM/Ogg/MP3/MPEG-TS muxer timing+layout and accurate-trim frame-window core; the **WebCodecs/GPU codec-seam** ops (lossy `decode`/`encode`, `convert` re-encode, accurate `trim`, `seek`, GPU filters, image pixel decode, raw PCM `AudioData` wrapping, and ADTS AAC→WAV native decode) run on the browser and are validated there; the **wasm-tail decode cores** (Vorbis/AAC/MP3, real vendored Symphonia) are Node-validated via clean-process decode oracles, while the public driver stream is browser-reachable only when `EncodedAudioChunk`/`AudioData` exist and the assets are co-vendored (ADR-042/069). **Routing/runtime update (ADR-006/020):** WASM codec loaders resolve `baseline` vs `isolated-simd-threads` at core-instantiation time, never during support probing; router cost buckets let tiny stages prefer cheaper native work over GPU/WASM setup where the driver supports the same operation. **Worker offload + ABR pool (ADR-087):** heavy `convert`/`trim` now run off the main thread when `createMedia({ worker })` selects offload — the engine serializes the **op** (input `ArrayBuffer` transferred + options minus `sink`), not the pipeline, and only encoded **bytes** stream back (no `VideoFrame`/`AudioData` crosses the boundary), with `{ pool:N }` fanning an ABR ladder across N reused epoch-tagged bridges; any worker-less/unsupported environment (Node, CSP-blocked module workers, no worker WebCodecs) downgrades to the **byte-identical** inline path (proven by a `convert(wav→wav)` byte-identity oracle driven through the full host↔worker loop in Node with the Worker mocked, including `force-software`). The worker boot + pool stay off the eager static closure (a separate `dist/worker.js` entry + a lazy host chunk); the `longtasks≈0` proof is the browser `performance` family.

## probe / metadata

- **Goal:** read container/track/codec/duration/tags without decoding.
- **Ladder:** TS header reader (range/bytes). No fallback needed; **never `<video>`**.
- **Why [data]:** mediabunny won 32/51 probe via a cheap `getDurationFromMetadata` header read; `platform` (HTMLMediaElement `loadedmetadata`) was 600–7000× slower. remotion-media-parser/mp4box also win here with pure-JS header parsing.
- **Flow:** `range(0, N)` the head → first sniff supported image magic (GIF/PNG/JPEG/WebP/AVIF) and, on a match, run the pure image header parser → otherwise route to a container driver and prefer its optional metadata-only `probe(src)` hook, falling back to `demux(src).tracks`. MP4/MOV and WebM/MKV ship metadata hooks so probe does not construct live packet streams, materialize MP4 packet byte tables, or build WebM frame arrays. Container probe parsers read boxes (`ftyp/moov/mvhd/stsd`), EBML (WebM/MKV), `OggS` (Ogg), `fLaC` (FLAC), `RIFF…WAVE` (WAV) / `RIFF…AVI` (AVI), `FORM…AIFF`/`AIFC` (AIFF), `caff` (CAF), MP3 frame-sync / ID3, ADTS sync, MPEG-TS `0x47` packet sync, HLS `#EXTM3U` playlist → `MediaInfo`. Read only what's needed for containers where the metadata grammar allows it (moov may be at the tail → one extra range read; MP4 reads the `stsz` count without expanding sample sizes; headerless WebM may still scan Cluster timing to preserve strict duration/fps metadata); images are small, self-describing byte payloads and are probed as images before the generic ISO-BMFF `ftyp` route so AVIF is not mistaken for MP4. The image header path extracts frame count, loop count, and exact GIF/APNG/WebP per-frame delay sums when those headers store timing. All 12 container drivers plus image probe are Node-validated against the real corpus (golden-metadata + ffprobe/sips-style truth goldens).
- **Edge cases:** moov-at-end MP4, AVIF `ftyp` brand preemption, animated GIF/APNG/WebP frame counts and header-delay sums (including zero-delay frames without renderer minimum-delay clamping), JPEG SOF marker walk, headerless `MediaRecorder` WebM, VFR, rotation matrix, garbled ID3/ilst tags, multitrack, CENC (read clear metadata without touching encrypted mdat).
- **Oracle:** `golden-metadata` + image-header goldens — exact container/codec/dims, exact image frame counts/delays, and ±1-frame duration for timed media containers.

## demux

- **Goal:** container → per-track encoded packet streams + `TrackInfo`.
- **Ladder:** TS streaming demuxer → WASM demuxer (exotic containers).
- **Why [data]:** mediabunny 19/43, ffmpeg.wasm 10 (mostly where it was simply present); pure-TS wins on latency/bundle/no-COOP.
- **Flow:** stream bytes → parse container → emit `EncodedVideoChunk`/`EncodedAudioChunk` per track (lazy, per-track `ReadableStream`). MP4/MOV applies supported single-rate edit-list `media_time` offsets only at this packet/WebCodecs seam, so packets expose the source presentation timeline while same-family stream-copy keeps raw `stts`/`ctts` ticks.
- **Edge cases:** B-frame DTS↔PTS, MP4 edit-list preroll offsets, VFR timing, multitrack selection, MPEG-TS PTS wraparound, HLS segment sequences, AAC-ADTS/MP3 frame sync; **graceful** on truncated/zeroed/header-destroyed/zero-length.
- **Oracle:** `golden-packets` — exact packet count, sizes, keyframe flags, 0µs PTS drift.

## decode + seek

- **Goal:** encoded packets → `VideoFrame`/`AudioData`; seek to a time.
- **Ladder:** WebCodecs **hardware** → WebCodecs **software** → WASM decoder.
- **Why [data]:** the WebCodecs trio (platform/mediabunny/web-demuxer) own decode-seek; hardware is fastest. Bit-exact decode wins are **GPU/platform-specific** (M1 ANGLE) → use `force-software` for cross-machine determinism.
- **Flow (seek):** find the keyframe at/just-before the target → `decoder.decode()` from there → discard frames before the target PTS → emit. Honor backpressure ([`06`](06-execution-and-runtime.md) §3).
- **Edge cases:** open-GOP, B-frame reorder, **VFR seek lands on the true PTS**, 10-bit (High10), VP9 alpha, 4K, 1×1/2×2 tiny dims, extreme fps (1, 240).
- **Oracle:** `decoded-frames-bitexact` (sha256 vs golden) where deterministic; else `ssim-psnr` with `exactFrames>0` (we reject `exactFrames==0`-only passes, ADR-018).
- **As built (ADR-026/030/049/057/063/069/080):** `WebcodecsVideoDriver`/`WebCodecsAudioDriver` (`tier:'hardware'`, codec-agnostic by config) are the decode tier for compressed tracks; each coder is a `TransformStream` (configure on `start`, decode per `transform`, `flush()` on close). **No live B-frame reorder buffer** — `VideoDecoder` emits in presentation order per the W3C spec (the UA reorders), so the driver enqueues in arrival order; the pure `reorderByTimestamp`/`isPresentationOrdered` helpers are for *captured*-stream tests/tooling only. Hardware-first under determinism (`auto → 'prefer-hardware'` video / `'no-preference'` audio; `force-software → 'prefer-software'`). Backpressure awaits `dequeue` while `*QueueSize ≥ 8`. **`decode()`** returns the `{ video, audio }` frame streams *lazily* (the demux + codec routing runs on first pull; an undecodable track yields an empty stream, not an error). Bare Matroska/WebM HEVC (`hevc`/`h265`) is normalized from `hvcC` `description` bytes into an exact `hev1.*` RFC-6381 string before WebCodecs probing; MP4/MOV `hvc1.*`/`hev1.*` strings pass through unchanged, so 8-bit and 10-bit/HDR streams are decided by the browser's exact `isConfigSupported` verdict. There is no bundled software HEVC decoder tail in this build, so an unsupported HEVC profile/level/browser combination is an honest `CapabilityError`, not a fabricated fallback. Raw PCM WAV/AIFF/CAF audio tracks bypass the fictional WebCodecs `pcm-*` codec route: their container driver parses canonical `PcmAudio` through `decodePcmAudio`, and the engine wraps it as pull-driven `f32-planar` `AudioData` chunks (4096 frames/chunk, sample-index timestamps) when `AudioData` exists; Node/unsupported browsers raise a typed capability miss before reading the whole source. The real Symphonia WASM audio tails are auto-registered but have the same host-frame boundary: their public `supports()` requires `EncodedAudioChunk` and `AudioData`, while the cores themselves remain Node-validated directly. Protected MP4 CENC/CBCS tracks keep probe/demux/decrypt metadata and original codec config for metadata, but `TrackInfo.encrypted` makes generic decode/seek reject with a typed `MediaError` until an explicit decrypt path has produced clear samples, so ciphertext is never handed to WebCodecs as if it were clear media. For supported images, the same lazy contract bypasses demux and returns an `ImageDecoder`-backed video stream; the paired audio stream is empty. Node has no `ImageDecoder`, so image decode raises a typed `CapabilityError` there rather than fabricating pixels. **`seek()`** scans packets for the last keyframe at/before the target (`startAtSeekKeyframe`), decodes from it, and `seekFrame` drops+`close()`s every earlier frame, returning the first at/after the target (caller-owned); seeking past EOF returns the closest (last) frame. Decoder output is owned by the readable consumer (closed once by it). Browser-validated (ADR-025); the pure helpers and image header parser are Node-tested.

## encode

- **Goal:** raw frames → encoded chunks.
- **Ladder:** WebCodecs **hardware** → WebCodecs **software** → WASM encoder (codecs WebCodecs can't encode).
- **Why [data]:** remotion-webcodecs/mediabunny win on `encodeFps`; ffmpeg.wasm libx264/libopus is the fallback where WebCodecs lacks an encoder, but ~20–35× slower.
- **Flow:** configure encoder (codec/bitrate/crf/keyframe interval) → feed frames → emit chunks + a decoder config (`description`) for the muxer.
- **Edge cases:** gapless priming (AAC/Opus), fps up/down-conversion, CRF/two-pass/bitrate modes, forced keyframe interval, alpha (VP8/VP9), HDR.
- **Oracle:** re-decode the output and compare `ssim-psnr` (+ `playback-smoke`); structural `reference-reimport`.
- **As built (ADR-026/029/030/051/080/084):** the encoder is a `TransformStream<RawFrame, EncodedChunk>` that **consumes** each input frame — `encode()` then `close()` in a `finally`, so the frame closes exactly once even on throw/abort (doc 06 §3). The public-token→codec-string + `*EncoderConfig` mapping is pure (`codec-pipeline.ts`: `buildVideoEncoderConfig`/`buildAudioEncoderConfig`, default profiles e.g. `avc1.42E01E`, `hev1.1.6.L93.B0`, `mp4a.40.2`; an omitted codec preserves the source string). For the public `h264` token, `h264LevelIdcForDimensions` still computes the minimum Annex-A level from output macroblocks and fps, while `h264CodecStringForDimensions` floors the browser-facing encode string at L3.0 for tiny MP4 playback/seek compatibility and scales upward for 720p/1080p/4K (ADR-084). Preserved HEVC Main (`profile_idc=1`, e.g. 8-bit `hvc1.1.*`/`hev1.1.*`) is passed to WebCodecs exactly; preserved HEVC Main10/non-Main (`profile_idc!=1`, e.g. `hev1.2.4.L93.90`) raises a typed `CapabilityError` before muxing because this build has no software HEVC encoder tail and must not silently downconvert HDR/10-bit output. **Encoder→muxer config bridge (ADR-029/051):** the encoder publishes its `DecoderConfig` (codec string + `description`) on the first chunk's metadata; the additive, contract-untouched `VideoEncoderStageOptions.onDecoderConfig` / `AudioEncoderStageOptions.onConfig` sink hands it to the engine, which allocates the muxer track lazily on the first chunk (`drainEncoderToMuxer`). When convert/transcode is re-encoding a source track, that same `TrackInfo` carries the source `durationSec` so muxers preserve the declared duration instead of extending output to encoder packet-tail padding. `keyFrameInterval` rides the same options object (GOP via `shouldKeyframe`). **`encode({})`** validates shape before muxing: no streams, or a stream with no matching target → `InputError` (after cancelling the unconsumed stream so no frame leaks); a non-chunk-muxable target (WAV) → `CapabilityError` (ADR-030). Browser-validated (ADR-025); the config/drain helpers are Node-tested with fake chunks.

## mux

- **Goal:** encoded packets + layout → a container byte stream.
- **Ladder:** TS muxer (MP4/WebM/Ogg/WAV/ADTS/TS) → WASM (containers not worth hand-writing).
- **Why [data]:** mediabunny dominates mux **46/52** — pure-TS muxers, `StreamTarget`, faststart, fragmented.
- **Flow:** `addTrack` per stream → `write` packets (preserving PTS/duration and B-frame `ctts`) → `finalize` (write `moov`/index). faststart = moov before mdat; fragmented = CMAF segments.
- **Edge cases:** B-frame composition offsets, multitrack, faststart-reserve vs in-memory, **illegal codec-in-container must reject** (e.g. H.264→WAV, VP9→ADTS), zero-track empty input → clean reject.
- **Oracle:** `reference-reimport` (re-demux output → exact tracks/packets/duration) + `mp4-box-layout` (moov/mdat order).
- **As built (ADR-028/037/048/051/064/065/066/067/070/071/072/079/081/097/098/113):** the `EncodedChunk`-seam muxers are real for **MP4/MOV** (`Mp4Muxer` over `writeMp4`), **WebM/MKV** (`WebmMuxer` over `writeWebm`), **Ogg** (`OggMuxer` over `writeOgg`), **MP3** (`Mp3Muxer` with Xing/VBR repair), and **MPEG-TS** (`MpegTsMuxer` over `ts-write`, H.264/AAC only). Public `media.mux()` now routes directly to the target container's chunk muxer: callers pass `PacketStreams` descriptors (`{ track, packets }` per media type), the engine validates that each `TrackInfo` matches the slot and carries codec configuration, drains the caller's `Packet | EncodedChunk` streams through `drainEncoderToMuxer`, finalizes, and materializes the chosen sink. Bare packet streams without `TrackInfo` are rejected with `InputError` and cancelled, because the muxer cannot infer codec-private boxes/headers, dimensions, audio layout, media type, or declared duration from chunks alone. Each generic muxer buffers track chunks in decode/arrival order and serializes on `finalize` (`output` is a one-chunk `ReadableStream` unless a fragmented path is requested — `Mp4Muxer({ fragmented:true })` CMAF MP4, ADR-046, or `WebmMuxer({ fragmented:true })` init-segment + live Cluster-per-fragment CMAF WebM, ADR-091 — which enqueue each segment separately for incremental `StreamTarget` writes). For large cross-container WebM/MKV **remux**, the engine does not use the buffer-all `WebmMuxer`: it routes to `WebmStreamingMuxer`, registers all tracks up front, schedules one packet per selected source track by DTS, and emits an unknown-size Segment plus bounded top-level Clusters as packets are written. **Codec-private boxes/headers:** MP4 maps AVC/AAC by synthesizing `avcC`/`esds` from `description`; for bare `h264` sources it accepts Matroska-style `avcC` descriptions directly, or synthesizes `avcC` from Annex-B SPS/PPS and converts access units to length-prefixed AVC before writing; for bare `aac` sources it accepts Matroska/WebCodecs AudioSpecificConfig descriptions directly, or derives ASC from ADTS headers and strips ADTS framing before writing raw AAC access units. MP4 carries HEVC/FLAC descriptions verbatim as their config boxes, synthesizes AV1 `av1C` and VP9 `vpcC` from validated codec strings, converts OpusHead to `dOps`, and maps MPEG Layer III frames into MP4 as `mp4a.6b` with ESDS while preserving MP3 packet bytes. A codec without derivable or supplied private data (for example HEVC without `hvcC`) is still a typed `CapabilityError` rather than a sample entry with missing private data. WebM/MKV maps codec strings to Matroska `CodecID` + `CodecPrivate` (including AAC ASC when present), and the EBML DocType follows the requested target (`webm` vs `matroska`) so MKV outputs parse as `mkv`. Ogg writes Opus, Vorbis, and FLAC audio: OpusHead for Opus, Vorbis' Xiph-laced id/comment/setup headers from `description`, and the official Ogg-FLAC mapping header from native FLAC metadata (`fLaC` + STREAMINFO + metadata blocks). MPEG-TS writes PAT/PMT PSI tables, one program, continuity counters, PCR on the video PID when present, PES PTS/DTS in the 90 kHz clock, AVCC→Annex B H.264 with SPS/PPS before keyframes, and raw AAC→ADTS frames from ASC/sample-rate/channel metadata. Unknown or illegal codec/container pairs are typed `CapabilityError`s, never malformed output. **Timing:** MP4 reconstructs DTS/`ctts`; MP4/MOV demux applies supported edit-list `media_time` offsets to packet PTS/DTS before cross-container muxing; WebM/MKV stores PTS blocks in decode order using `Packet.dtsUs` and honors a declared track duration over packet-tail padding; Ogg stamps cumulative granule positions (Opus from the packet TOC before host duration hints, with declared final duration allowed only as an in-packet trim; Vorbis from exact packet durations when present, or from scaled packet weights anchored to a declared final duration when container packet cadence is approximate/laced; FLAC by exact decoded sample counts); MP3 writes frame cadence plus a fresh Xing frame for VBR metadata; MPEG-TS emits each access unit in DTS order, accepts finite signed packet PTS/DTS, rebases only negative preroll to a non-negative PES clock, and preserves positive same-container TS start times. Only `write()`'s `EncodedChunk.copyTo` byte extraction is browser-guarded; the timing + serialization models are Node-validated through pure `addChunkStruct`/writer tests and public mux real-corpus tests.

## remux

- **Goal:** container → container, **stream-copy** (no re-encode).
- **Ladder:** demux(TS) → copy packets → mux(TS); WASM only for exotic containers.
- **Why [data]:** mediabunny 25 / ffmpeg.wasm 22 — close; pure-TS wins latency + no-COOP, ffmpeg wins where libav's container breadth helps.
- **Flow:** `demux` → pass `EncodedChunk`s straight into `mux` (no codec stages) → faststart optional.
- **Edge cases:** codec/container compatibility (codec must be legal in the target), rotation/metadata survival, headerless/truncated → graceful.
- **Oracle:** `reference-reimport`. **Anti-cheat note:** a real remux re-lays-out the container; do **not** ship the SUSPECT shortcut of flipping `ftyp` bytes and returning the input (ADR-018).
- **As built (ADR-021/045/048/052/056/064/065/066/067/068/071/072/098/113):** same-family MP4/MOV and MPEG-TS remux use pure driver-native `streamCopy` when every source track is requested. MP4/MOV stream-copy validates sample byte ranges up front, coalesces URL-backed sample reads into bounded range windows, and preserves raw DTS/`ctts`, keyframe flags, codec-private data, and output layout. MPEG-TS stream-copy parses PAT/PMT/PES once, selects the original H.264 Annex-B and AAC ADTS access units, then feeds those bytes back through the TS writer so PES PTS/DTS, keyframe flags, and AU boundaries survive without WebCodecs. The MP4 demuxer exposes an optional payload-free `packetTable()` for non-fragmented sample-table inputs, so benchmark/golden-packet consumers can enumerate packet size/PTS/DTS/keyframe metadata without reading `mdat`; supported MP4 edit-list `media_time` offsets are applied to those packet timestamps and to the live `packets(trackId)` seam, not to the raw stream-copy sample tables. The actual payload stream remains available and also reads sample bytes through bounded range windows. Cross-container remux normally uses the packet seam: demux source tracks with configs, optionally apply single-source selectors such as `audio:0`, and feed edit-list-adjusted PTS plus `Packet.dtsUs` into the target chunk muxer. For WebM/MKV targets that request fragmented/live output or whose known source size exceeds the buffer-all remux ceiling, the engine switches to the streaming packet scheduler instead: it opens one reader per selected track, keeps only the next packet from each reader, writes packets in DTS order to `WebmStreamingMuxer`, and flushes bounded top-level Clusters as video keyframes, timecode-span bounds, or the block cap require. That lifts the former GB-scale MP4→MKV safety decline without raising memory caps or hardcoding fixture ids. Track selection is intentionally applied before muxing so audio-only Ogg authoring from a video+audio WebM copies just the selected Vorbis stream; a selector that matches nothing is a typed `InputError`, not a silent all-track mux. Native FLAC→Ogg is a packet-preserving version of that same path: the FLAC driver enumerates exact native frame byte spans and carries native metadata in `config.description`, while `OggMuxer` lays those frames into Ogg-FLAC packets without decoding or fabricating seektable repair. H.264/AAC→MPEG-TS uses the same packet seam, with the TS muxer converting MP4-style H.264 samples to Annex B and AAC payloads to ADTS only when needed, while rebasing negative edit-list/B-frame preroll to a valid non-negative PES clock. Bare-H.264 and bare-AAC sources such as MPEG-TS and Matroska can also remux to MP4/MOV: `Mp4Muxer` uses `config.description` when the source carries `avcC`/ASC, derives `avcC` from Annex-B SPS/PPS for H.264, and derives ASC plus raw AAC access units from ADTS-framed AAC. WebM AV1/VP9 with Opus and elementary MP3 sources now use the same packet seam: MP4 synthesis writes AV1 `av1C`, VP9 `vpcC`, Opus `dOps`, or MP3 ESDS records only from source codec strings/headers that normatively contain the fields, preserving packet bytes and timestamps without re-encode.

## trim

- **Goal:** cut a time range.
- **Ladder:** keyframe-aligned **packet-copy** (fast, lossless) or **frame-accurate** (decode from a safe preroll, trim decoded frames at presentation timestamps, re-encode, mux).
- **Why [data]:** mediabunny 31/42 — pure-TS packet-copy; ffmpeg.wasm for some audio copy paths.
- **Flow (accurate):** validate range against real duration → demux → video starts at the last keyframe at/before `start`, audio starts at the head → decode → keep frames with PTS in `[start,end)` → close every preroll/end-boundary frame → rebase the first kept frame to `0` → re-encode each kept stream → drain encoded chunks into the source-family muxer.
- **Edge cases:** B-frames, open-GOP, VFR, FLAC `STREAMINFO`/seektable repair, start/end past EOF, inverted/zero-length range, bitflipped/truncated source → graceful.
- **Oracle:** `trim-boundaries` (duration within tolerance) + boundary-frame digests for `accurate` (we prefer the stronger gate, ADR-018).
- **As built (ADR-021/043/047/068/082/096/097/114):** MP4/MOV keyframe trim is driver-native stream-copy with container-bounds validation and browser AVC decode preflight where available. The ordinary public keyframe-trim path now passes the same sink-sensitive `buffered`/`streaming` hints as full remux: selected samples are represented as layout-only records, the MP4 writer plans `ftyp`/`moov`/`mdat` from byte lengths and timing, and payload bytes are copied from bounded source range windows only for the kept GOP/audio span. When browser AVC validation is available, the preflight also reads the selected GOP from source windows instead of forcing a full selected-sample materialization. MPEG-TS keyframe trim is also driver-native for same-container outputs: the driver computes the public trim window against the earliest source PTS (transport streams often start at a nonzero timestamp), starts video at the last keyframe at or before `start`, starts audio at the first overlapping ADTS frame, stops on decode time before `end`, and reserializes only the selected original AUs through the TS writer. Native FLAC same-container trim is sample-domain and lossless: the FLAC driver decodes to canonical PCM, slices `[start,end)` before any transform, and re-authors FLAC with STREAMINFO total-sample/MD5 repair, so seektable and no-seektable inputs land on the same decoded PCM. MP3, ADTS/AAC, and Ogg/Opus audio-only keyframe trims use the packet seam: keep whole compressed packets overlapping `[start,end)`, rebase the first kept packet to zero, and let the existing muxers repair Xing/ADTS/Ogg timing metadata. Accurate trim now routes to the same browser codec seam as convert: route a decoder for the first decodable video/audio tracks, decode video from the seek keyframe preroll (audio from the stream head), pass decoded frames through `trimTimedFrameStream`, re-encode, and mux with the source family's chunk muxer. The frame-window helper is strict about ownership: frames before `start` and the first frame on/after `end` are closed immediately, upstream is cancelled at `end`, replacement restamped frames close their source exactly once, and emitted frames stay open for the encoder to close. The Node oracle covers boundary inclusion/exclusion, adjacent-window additivity, unchanged-frame ownership, restamp-failure cancellation, MP4 selected-range reads, and the real MP4 public call now reaches the codec router rather than the old stub; live decoded-pixel validation remains in the browser harness under ADR-025. Unsupported target families, encrypted tracks, missing WebCodecs, and malformed trim ranges are typed misses/errors, never passthrough.

## convert (the headline op)

- **Goal:** produce a target container/codecs, applying filters; **auto-route** copy-vs-re-encode per stream.
- **Ladder:** per stage — demux(TS) → decode(WebCodecs hw→sw→WASM) → filter(GPU→WASM) → encode(WebCodecs hw→sw→WASM) → mux(TS).
- **Why [data]:** mediabunny 50/84 transcode via WebCodecs decode/encode + streaming mux; ffmpeg.wasm 19 where codecs/filters are browser-missing; remotion-webcodecs 9.
- **Auto-route:** if a stream already matches the requested codec/params → **stream-copy** (remux path); else re-encode. The developer doesn't choose (ADR-012).
- **Edge cases:** resize/crop/pad/rotate/flip, fps & bitrate/CRF/two-pass, colorspace 709↔2020, HDR→SDR tonemap, audio downmix/resample, container change, VP9 alpha keep, 10-bit; the 3 no-winner cases (FLAC source, HEVC-10-bit out, VP8 playback-smoke) [data: Finding 8].
- **Oracle:** `ssim-psnr` **with `exactFrames>0`** + mandatory `playback-smoke` (the benchmark's `h264_to_vp8` produced output that failed playback — we treat that as a fail). Strengthen to bit-exact for lossless/copy paths.
- **As built (ADR-026/027/028/037/051/062/065/074/075, routing in `engine.ts`):** `convert` tries paths in cost order. **(1) PCM-native / compressed-to-WAV** (ADR-022/024/050/054/059/061/074/075): a PCM/no-codec audio target routes to the source container's `transformPcm` for raw PCM (WAV/AIFF/CAF) or `decodePcm` for compressed sources that can author WAV (FLAC pure TS; ADTS AAC via native WebCodecs with `wasm-aac` fallback) — never through a fake WAV/AIFF/CAF chunk muxer. `PcmTransform.container` names the raw-PCM output wrapper, so WAV↔AIFF↔CAF conversion stays in the TS PCM path after the source driver parses its own bytes. Explicit PCM codec tokens (`pcm-u8`, `pcm-s8`, `pcm-s16`, `pcm-s24`, `pcm-f32`, `pcm-s16be`, etc.) select the target raw sample format/endianness; an omitted/generic PCM target preserves the source wire format as far as the requested wrapper permits, including value-preserving `s8↔u8` conversion when moving between signed-8 AIFF/CAF and unsigned-8 WAV. PCM-native `audio.gainDb`, `audio.fade`, `audio.biquad`, and `audio.dynamics` run in the shared transform helper, ordered as gain/fade → remix/resample → biquad chain → dynamics normalize/limit so limiters see the final sample-rate/EQ shape. **(2) Lossless stream-copy** (ADR-021/012): a pure container change with no re-encode (`isPureStreamCopy` — no dropped track, no video filter/color op/codec/dims/fps/bitrate change and no audio codec/rate/channel/bitrate/gain/fade/dynamics/biquad change) routes to `streamCopy` when the source container supports the target. **(3) Codec seam:** otherwise demux → per-track decode (WebCodecs hw→sw) → optional filter chain (video — `videoFilterSpecs` orders **crop→resize→rotate→flip→colorspace→tonemap**, ADR-027/032/038/062) → encode → the requested chunk muxer (MP4/MOV, WebM/MKV, Ogg, or MPEG-TS when the encoded codec is legal there). The output container must be chunk-muxable or it is a typed `CapabilityError`; declared source durations are carried into the mux track when available so duration-preservation oracles do not depend on encoder packet-tail quirks. The auto-route decision is the developer's only invisible choice (ADR-012). The cheap routing predicates live in eager `codec-routing.ts`; heavier config normalization, encoder config, packet drain, and seek helpers stay behind lazy `codec-pipeline.ts` imports from live codec paths (ADR-103). Both sets are Node-tested; the live decode→filter→encode→mux is browser-validated (ADR-025).

- **Gain/fade/dynamics/EQ updates (ADR-060/061/074/089):** public `audio.gainDb` is routed through both audio paths. PCM-native WAV/AIFF/CAF targets pass gain, `audio.fade`, `audio.dynamics`, and `audio.biquad` into `PcmTransform`; codec-tier re-encodes now emit the full **gain → fade → remix → resample → biquad → dynamics** `AudioData` filter chain before the encoder, so fade/dynamics/biquad cross the lossy codec seam too (ADR-089: stream-stateful stages — persisted biquad registers, duration-aware fade tail, non-causal normalize buffer — bit-exactly equal to the whole-signal/PCM-native result). They are no longer codec-seam capability misses.

## audio-dsp

- **Goal:** PCM format/endianness conversion, gain, fade, channel up/down-mix, resample, lossy encode.
- **Ladder:** TS for format/gain/mix/fade **and resample** (pure-TS band-limited windowed-sinc) → WASM for lossy encode. (A WASM **soxr** / WebAudio `OfflineAudioContext` resampler remains a possible future alternative, but is **not** needed — the TS resampler ships in-tier.)
- **Why [data]:** ffmpeg.wasm won audio-dsp 25/36 — but ~most are PCM format/endianness/gain/mix/fade, which are kilobytes of TS; **resample** is a few hundred more lines of exact TS (windowed-sinc), so only **lossy encode** genuinely needs WASM (Finding 4). We reclaim the cheap majority *and* resample in TS.
- **Flow:** parse PCM → transform samples (TS) → re-serialize. For raw-PCM containers (WAV/AIFF/CAF) the engine takes the **PCM-native path** (ADR-022/054/059/061/074/075): `ContainerDriver.transformPcm` runs the TS kernels (`src/dsp`) in order **gain → fade → remix → resample → biquad/EQ → dynamics normalize/limit** and re-serializes to the requested raw-PCM wrapper plus sample-format/endianness when the public target names them, otherwise preserving the source wire format where legal for the target wrapper. WAV 8-bit PCM is unsigned (`pcm-u8`); AIFF/CAF 8-bit integer PCM is signed (`pcm-s8`), so no-codec cross-wrapper conversion maps between the two through canonical samples, while explicit incompatible 8-bit requests reject with `CapabilityError` rather than mislabeling bytes. The shared `writePcmContainer` writer keeps WAV/AIFF/CAF output ownership inside first-party PCM drivers instead of asking the EncodedChunk muxer to author raw samples. The resampler uses a cached rational-rate polyphase bank for ordinary integer sample-rate pairs (ADR-058), falling back to the dynamic dense sinc table only for unusual ratios with too many phases. For compressed sources that can author WAV directly, `ContainerDriver.decodePcm` decodes to canonical PCM and then applies the same transform order: FLAC is pure TS (ADR-024), ADTS AAC is browser-native WebCodecs with `wasm-aac` fallback (ADR-050). `convert(→ wav/aiff/caf)` channel mix / format / **sample-rate**, PCM-native fade, dynamics, and biquad/EQ all ship here for raw-PCM sources; only lossy encode raises `CapabilityError` until the WASM tail.
- **Public gain/fade/dynamics/biquad (ADR-060/061/074):** `AudioTarget.gainDb` is the canonical public gain unit. A zero gain is a no-op; non-finite values are rejected before filter construction. Harness-only `gainLinear` values are converted to dB in the browser adapter only when positive and finite, keeping the engine surface one-unit and typed. `AudioTarget.fade` exposes PCM-native fade-in/fade-out durations in seconds plus a `linear`/`equal-power` curve. `AudioTarget.dynamics` exposes one optional peak/RMS normalize step plus one optional hard/soft limiter, all in dBFS. `AudioTarget.biquad` accepts one RBJ-cookbook section or a readonly section chain for PCM-native EQ/filtering. Invalid durations, dB values, limiter modes/knees, and out-of-band biquad frequencies reject with typed errors, and the codec seam refuses fade/dynamics/biquad rather than silently dropping them.
- **Edge cases:** unsigned WAV `u8`, signed AIFF/CAF `s8`, s16/s24/f32, LE/BE, BS.775 downmix coefficients, upmix, variable channel count, empty audio, 5.1; resample of any ratio (44100↔48000, arbitrary target), equal-rate identity copy, zero-extended edges.
- **Oracle:** `decoded-audio-pcm` (sample-exact) for the lossless conversions (format/gain/mix); for **resample** — a band-limited (lossy) filter — `property-invariant` (frame-count/channels) plus a round-trip SNR floor; we push toward PCM-exact where the math is deterministic.
- **Compressed-source WAV extraction:** **FLAC decode** is **pure TS** (ADR-024, `codecs/flac`), not the WASM tail: lossless integer Rice + fixed/LPC + decorrelation, exposed via `ContainerDriver.decodePcm` (FLAC → WAV) and validated **bit-exact** on the IETF FLAC conformance corpus via STREAMINFO MD5. **ADTS AAC extraction** uses `AdtsDriver.decodePcm` (ADR-050): native `AudioDecoder` first, vendored `wasm-aac` fallback, canonical PCM concatenation, then WAV `pcm-s16`. **Resample** is likewise pure-TS now (ADR-022); only lossy **encode** stays WASM-tail.
- **As built — the full pure-TS DSP kernel set (`src/dsp`, Node-validated):** beyond format-convert / `gain` / `remix` (BS.775) / `resample` (windowed-sinc, cached rational-rate polyphase hot path), the kernels also include **`fade`** (`fadeIn`/`fadeOut`/`crossfade`, with `FadeShape` curves), **`dynamics`** (`limit`/`normalizePeak`/`normalizeRms`, `LimitMode`), and **`biquad`** (RBJ-cookbook `designBiquad`/`biquad` parametric EQ + `magnitudeResponse`/`polesInsideUnitCircle` stability checks) — all pure, deterministic, sample-tested in Node. **Wiring status:** `gain`/`remix`/`resample` are reachable via the public ops (container `transformPcm` for raw-PCM, and the `audio-dsp-filter` `AudioData` seam, ADR-033); **`fade`/`dynamics`/`biquad` are reachable for PCM-native convert/decodePcm outputs** (ADR-061/074) **and now also across the lossy codec `AudioData` seam** as stream-stateful stages (ADR-089: persisted biquad `z1`/`z2`, duration-aware fade tail, non-causal whole-signal normalize buffer — validated bit-exact in Node against the whole-signal kernels in arbitrary chunk splits).

## filters (video)

- **Goal:** pixel transforms — resize, crop, pad, rotate, flip, colorspace, tonemap.
- **Ladder:** WebGPU → Canvas2D → native CPU → WASM (libavfilter). (The **WebGL rung is omitted**, ADR-027 — Canvas2D `drawImage` is GPU-accelerated and exact for the geometric ops, so it is the single fallback before the pure-TS CPU floor.)
- **Why:** GPU shaders run these faster than software libavfilter **and ship zero bundle** — reclaiming a chunk of ffmpeg's "coverage" for free (Finding 5/6). Geometric ops (flip/rotate90/crop) can be exact.
- **Flow:** `VideoFrame` → upload to GPU texture (or `OffscreenCanvas`) → shader → new `VideoFrame` (close the input).
- **Edge cases:** odd dimensions, fit modes (contain/cover/fill), 90/270 dim-swap, BT.709↔2020 matrices, PQ/HLG→SDR tonemap, alpha preservation.
- **Oracle:** `ssim` vs reference; exact digest for pure geometric transforms.
- **As built (ADR-027):** two `FilterDriver`s register — `webgpuVideoFilterDriver` (`substrate:'webgpu'`, primary) and `canvas2dVideoFilterDriver` (`substrate:'canvas2d'`, fallback); the **WebGL rung is omitted** (Canvas2D `drawImage` is GPU-accelerated and exact for the geometric ops). WebGPU samples one full-screen quad whose geometry is driven entirely by per-frame uniforms (`posScale`/`posOffset`/`uvScale`/`uvOffset`/`rot0`/`rot1`), so a single pipeline serves resize/crop/rotate/flip. Each filter is a `TransformStream<VideoFrame, VideoFrame>`; the renderer is built once on `start`, reused per frame, disposed on `flush`/abort. **Close-once:** each input `VideoFrame` is `close()`d in a `finally` right after the draw; a new output frame carries the source `timestamp`+`duration`. The geometry (`geometry.ts`: `Blit`/`Affine`, all dims integer ≥ 1, deterministic rounding) and uniform packing (`gpu-uniforms.ts`, std140, 48 bytes) are pure/Node-tested; the GPU/Canvas render is browser-validated (ADR-025). **Colorspace + tonemap are implemented (ADR-032)** via a second WGSL color pipeline that applies, per pixel, *decode-transfer → 3×3 linear-RGB gamut matrix → (tonemap operator) → encode-transfer*. The color science is pure/Node-tested in `gpu-uniforms.ts`: gamut matrices built from CIE xy primaries + D65 (reproducing the published constants bit-exactly — sRGB/709≡identity, the BT.601/709/2020 luma rows, 2020↔709), the sRGB / BT.709-2020 (BT.1886) / PQ (ST 2084) / HLG (BT.2100) transfer pairs (monotonic, black→0, SDR white→1, PQ peak→100, HLG peak→12 in SDR-white-relative linear units, round-trip ≤1e-5), an extended-Reinhard tonemap normalized to the source peak (black→0, peak→1, monotonic, Hable available), and the `parseColorSpace`/`planColor` spec→pipeline selector. `supports()` is honest per substrate: WebGPU does colorspace + tonemap for all targets; Canvas2D does colorspace only to the display space (srgb/bt709) and declines wider-gamut + all tonemap (→ router falls through). The source colorspace / HDR transfer is read from the live `VideoFrame.colorSpace`. The pixel render is browser-validated (ADR-025).
- **As built (ADR-038/076) — the CPU fallback (`cpu-video-filter`, the universal no-GPU floor):** a third `FilterDriver` (`substrate:'native'`, ranked below the GPU/canvas rungs and above WASM) that runs **all six** video ops in **pure TS** over `VideoFrame.copyTo(buf, {format:'RGBA', …})`, reusing the *same* validated math — geometry from `geometry.ts` and colour science from `gpu-uniforms.ts` (`eotf`/`oetf`/`applyMat3`/tonemap, the functions the WGSL shader mirrors). Because `copyTo`→`'RGBA'` returns the frame's pixels in the frame's **own** colour space (UA does only YUV→RGB, not display tone-management), the CPU path performs a **genuine** colorspace conversion to *any* target gamut and a **genuine** PQ/HLG→SDR tonemap — strictly more capable than Canvas2D for colour. So on a browser without WebGPU, wide-gamut colorspace + tonemap are no longer a miss — they route to this CPU driver. Close-once holds across the async `copyTo` (the source is fully read into the buffer before the await resolves, then `close()`d once in a `finally`); the output frame carries the source timing + an honest output `colorSpace`. The pure per-pixel transforms (`applyColorPlanToRgba`/`geometryToRgba` over a plain `RgbaImage`, no browser types) are **Node-validated to GPU parity** (≤1 LSB vs the independently-recomputed primitives — a falsifiable oracle); only the `copyTo`/`VideoFrame` glue is browser-only.

## encryption / decrypt

- **Goal:** CENC (`cenc`/CTR, `cens`/CTR-pattern, `cbcs`), HLS full-segment AES-128, and HLS TS SAMPLE-AES decryption given keys; clean rejection of unsupported schemes.
- **Ladder:** WebCrypto AES-CTR/CBC + TS ISO-BMFF box parse (`tenc/senc/saiz/saio`) → for HLS, AES-128-CBC over whole segments or SAMPLE-AES TS sample payload blocks.
- **Why [data]:** ffmpeg.wasm won `cenc-ctr`, mediabunny won `cbcs` (subsample pattern); both via real crypto, not canned output.
- **Flow:** parse protection boxes → derive per-sample IV/subsample map → `crypto.subtle.decrypt` → emit cleartext packets → optional remux. **`cenc`/AES-CTR, `cens`/AES-CTR-pattern, `cbcs`/AES-CBC-pattern, `hls-aes128`, and `hls-sample-aes` are implemented driver-native** (ADR-023/121): MP4 `ContainerDriver.decrypt` parses `enca`/`tenc`/`senc`, decrypts samples (whole-sample for audio, subsample-aware for video), and re-serializes cleartext. `cens` applies the `tenc` crypt:skip pattern to AES-CTR: only selected full 16-byte crypt blocks are gathered/decrypted/scattered, skip blocks and trailing partial blocks remain clear, and the CTR counter advances over encrypted crypt blocks only within the sample. `cbcs` applies AES-CBC over the protected ranges with the `tenc` crypt:skip block pattern (a per-protected-subsample CBC chain over the crypt blocks; skip blocks + trailing partial blocks stay clear; per-sample or `default_constant_IV`). `hls-aes128` (full-segment AES-128-CBC + PKCS#7, key+IV from `keys`) decrypts an MP4 segment as a unit or raw segment payload via `src/crypto/hls-aes.ts`. `hls-sample-aes` decrypts MPEG-TS H.264/AAC sample payload blocks in place: PAT/PMT identify the PIDs, PES payloads are reassembled, H.264 slice NALs keep the first 32 bytes clear then decrypt one 16-byte CBC block per 160-byte cycle, and ADTS AAC frames keep the first 16 bytes clear then decrypt the remaining full blocks. fMP4 SAMPLE-AES and SAMPLE-AES-CTR remain typed non-claims until real vectors exist. A scheme that contradicts the container's `schm` rejects as a typed mismatch.
- **Edge cases:** subsample patterns (`cens`/`cbcs`), per-sample IVs, `cbcs` constant IVs, TS PAT/PMT/PES boundaries, zeroed/bitflipped/truncated protection → graceful; **reject** unsupported schemes (`sample-aes-ctr`, `clearkey`, live EME acquisition) with a clear error, not a crash.
- **Oracle:** `decrypt-bitexact` — decoded frames sha256-match a cleartext golden. Browser-free, we use the **stronger** sample-level gate: an encrypt→decrypt round-trip on real media recovers the sample bytes **bit-exact**, for `cenc` (AES-CTR), `cens` (AES-CTR pattern), `cbcs` (AES-CBC pattern — 1:9, full 1:0, and a 5:5 cycle; whole-sample + subsample on real video bytes), `hls-aes128` (a `node:crypto`-encrypted segment, an independent oracle), and `hls-sample-aes` (real `hls_vod_000.ts` encrypted by a test-only Node AES-CBC SAMPLE-AES twin, then recovered byte-for-byte via playlist resolution and `media.decrypt`) (+ anti-cheat: cipher≠clear, wrong-key≠clear; + NIST AES-CTR **and** AES-CBC vectors on the crypto core).

## streaming-output

- **Goal:** produce output incrementally with bounded memory (live/streaming targets).
- **Ladder:** TS muxer with a `StreamTarget` (incremental writes) → faststart in-memory/reserve → fragmented CMAF.
- **Why [data]:** mediabunny 18/27 — `StreamTarget` incremental writes, TS continuity, headerless WebM live. (These were among the few **freshly-measured** wins, so the perf is trustworthy.)
- **Flow:** mux writes chunks to a `WritableStream`/OPFS as they're produced; never hold the whole output.
- **Edge cases:** TTFB targets, faststart variants, fragment boundaries, TS continuity counters, headerless WebM, massive inputs.
- **Oracle:** `reference-reimport` + duration-materialized property.
- **As built (ADR-034/046/052/053/091/099/100/101/102/113/114):** the streaming target now has both validated building blocks and public reachability. (1) **CMAF / fragmented-MP4 writer** — `src/drivers/mp4/fragment.ts`, a generator `fragmentMp4(tracks, opts)` that yields an **init segment** (`ftyp` + fragmented `moov` with empty sample tables + `mvex`/`trex`) and then one `moof`+`mdat` media segment per fragment, with monotonic `tfdt`, `trun` sample durations/sizes/flags/composition-offsets, `default-base-is-moof`, and keyframe fragment splits. `Mp4Driver.streamCopy({ fragmented:true })` and `Mp4Muxer({ fragmented:true })` route to that writer, so fragmented MP4 is reachable through the container paths that request it; full same-container MP4/MOV stream-copy into a `StreamTarget` is lazy on the source side too, and the driver also has a progressive `ftyp`/`moov`/`mdat` byte-stream plan that emits headers before payload and then range-reads compacted sample windows in track/sample order (`highWaterMark:0`, ADR-101). Keyframe MP4/MOV trim reuses that progressive layout planner for the selected sample range, so sustained copy trims can be source-bounded without rewriting the output oracle (ADR-114). **Fragmented/live WebM** is the parallel for WebM/MKV (ADR-091/099/113, `src/drivers/webm/ebml-write.ts`): `fragmentWebm` yields an **init segment** (EBML Header + an **unknown-size** `Segment` header + `Info` without `Duration` + `Tracks`) then one live top-level `Cluster` per fragment (video-keyframe / int16-timecode-span / block-cap boundaries off the *same* `buildBlockTimeline`), and `WebmMuxer({ fragmented:true })` enqueues each segment separately so a `StreamTarget` writes incrementally with peak output memory bounded to one Cluster. `WebmStreamingMuxer` is the true Cluster-on-write variant for large cross-container remux: it emits the same unknown-size Segment layout but never stores the whole packet timeline, applying backpressure after one queued segment and flushing Clusters before `finalize` when boundaries arrive. Validated Node-side: separate init+Cluster enqueues, an unknown-size (`-1`) Segment, no `Info/Duration`, keyframe-aligned fragments, an independent low-level block scan, a byte-exact golden, an ffprobe `nb_read_packets`-preserving reimport oracle (skipped loudly when ffprobe is absent), and Cluster-on-write tests that read a Cluster from the output before `finalize()`. (2) **`StreamTarget` sink** — `src/sinks/stream-target.ts`, part of the `Sink` union and exported as `toStreamTarget`, writes each produced chunk incrementally to a caller-owned `WritableStream<Uint8Array>` or a position-aware callback. The callback arm validates the destination shape up front (`WritableStream` or callback only), races source reads and callback writes against `AbortSignal`, maps unsupported destinations to `CapabilityError`, maps cancellation to typed `MediaError('aborted')`, and never waits on a stuck upstream cancel before surfacing the primary typed error. (3) **URL-backed MP4 stream-copy source reads** coalesce sample bytes into bounded HTTP range windows, so large static corpus inputs do not require whole-file browser reads or one range request per sample. For explicit ISO-BMFF buffer targets, ordinary outputs can use the MP4 driver's layout-only final-buffer fill from those same range windows, while GB-scale browser rows request fragmented MP4 and still return one final buffer so strict reference reimport stays stable without declaring stream telemetry (ADR-102). The browser harness adapter writes through `toStreamTarget`, reconstructs `MediaBytes.bytes` from observed positioned writes, maps ISO-BMFF `target:'stream'` size-ladder rows to lazy fragmented MP4 while keeping the progressive decode-equality invariant on progressive MP4, maps WebM-family `target:'stream'` rows to fragmented/live muxers, declares `headerless` after fresh Chromium/WebKit/Firefox live-layout passes, and declares `fastStart:reserve` only for the current suite's final moov-first layout/duration rows; sparse reserve backpatch telemetry remains unclaimed until the benchmark exposes it.

## robustness / negative

- **Goal:** never crash, never emit wrong output, on garbled/truncated/zeroed/bitflipped/empty inputs; correctly report unsupported.
- **Ladder:** the same op ladders, but the contract is **clean failure** — throw a typed `MediaError`/`InputError`, emit no output.
- **Why [data]:** mediabunny 31/60; image probe/decode rows are no longer an out-of-scope `NA` bucket for supported GIF/PNG/JPEG/WebP/AVIF inputs — they route through `ImageOps` (ADR-049), while malformed image bytes still reject cleanly.
- **Edge cases:** header-destroyed, tail-truncated, mid-file zeroed spans, bit-flips, zero-length, mislabeled container, 1×1 dims.
- **Oracle:** `graceful-failure` — **strengthened**: it is not enough to "not crash"; the engine must either produce a *correct* partial result or reject. Producing plausible-but-wrong output is a fail (ADR-018).
- **As built for parser fuzz (ADR-073):** corrupt fixture-derived inputs are gated by a typed-error contract, not by a console inventory. `src/test-support/fuzz/parser-robustness.test.ts` runs deterministic empty/truncate/wrong-magic/oversize/missing/duplicate/nested/random/bitflip matrices over MP4 `parseMovie`, MP4 `readMovie`, and the pure WAV/MP3/Ogg/FLAC/ADTS/AIFF/CAF/AVI/MPEG-TS/WebM parsers. Any raw `RangeError`/`TypeError`/hang is a failure; acceptable outcomes are a correct parse or a `MediaError` subclass. The parser surface now bounds MP4 reader reads/seeks, WAV `fmt `, Ogg page bodies and codec IDs, AIFF `SSND`, and AVI fixed header chunks before fixed-width `DataView` reads. The container benchmark includes a bounded seven-family `fuzz robustness` row so the strengthened path is performance-regression gated too.
- **As built for MP4 trim (ADR-043/047/114):** the pure stream-copy byte path rejects container-visible corruption before emitting output: sample byte ranges are bounds-checked, short reads reject, and malformed trim ranges are caught by the public API guard. In browsers, keyframe AVC trim adds a real `VideoDecoder` preflight over the selected GOP when WebCodecs supports the source config; output frames are closed immediately and the original samples are still written losslessly only after validation succeeds. The lazy selected-range trim path feeds that preflight from bounded source windows, not a full selected-sample array, so corruption validation and scale safety coexist. This catches entropy-coded H.264 payload bit-flips without filename checks. If WebCodecs/config support is absent, the engine honestly falls back to the container-only boundary rather than pretending it decoded.
