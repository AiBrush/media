## 3. THE TWO ROOT CAUSES (fix these and the list collapses)

Almost every deficit is one of two shapes. Diagnose each cell into A or B before touching code (parent §4 design note first).

### A — Whole-file processing where rivals seek to the index (`moov`/cues/idx) — the catastrophic + severe tail

- **Symptom:** our time scales with **file size/duration**; rivals are ~flat. `probe/perf-extract-metadata-massive` = us 17.5 s, mediabunny 44 ms; `demux/…massive…` = us 46 s, remotion 40 ms.
- **Cause:** we read/parse/materialize the entire stream to answer a question that only needs the header index. Rivals lazily parse `moov`/cues and **walk the sample table**, emitting packet records (offset/size/pts) without touching payload bytes.
- **Fix program:** make `probe`/metadata **O(1) seek-to-`moov`** (never a full scan); make demux/`iterate-packets` **lazily index-driven** — yield packet descriptors from the sample table, fetch payloads only on demand and only when the consumer/oracle requires bytes. Preserve exact packet semantics so the **same `golden-packets`/`golden-metadata` oracle still passes**.

### B — Fixed per-operation overhead — the 211-cell minor tail

- **Symptom:** on **tiny** inputs we are still 5–30× slower though the real work is microseconds. `mux/pcm_s16_to_wav` (a header + copy) = us 110 ms, mediabunny 4 ms; `probe/flac_seektable` = us 38 ms, rivals 2–7 ms.
- **Cause:** a large constant per call — module init, WASM instantiation, WebCodecs configure latency, worker spin-up/handshake, redundant buffer copies, no reuse/pooling.
- **Fix program:** profile the constant floor on a trivial op (`mux/pcm_s16_to_wav`), then amortize it: warm/pool WebCodecs configs and workers, reuse buffers (zero-copy `transfer` over copy), skip worker/WASM entirely for trivially small pure-TS work, lazy-init so a probe pulls nothing heavy. Every change stays behind the frozen public API and the budgets.

> **A third, rarer shape:** genuine algorithmic slowness in a hot loop (e.g. FLAC seektable parse, Rice decode). Fix with SOTA algorithm/data-structure work — never by doing less than the oracle validates.

## 4. FAIR-COMPARISON RULES (the honesty guardrails — read before claiming any win)

Speed work is where faking is tempting. These are hard constraints:

1. **Same oracle, same work.** A cell counts as won only if, after the change, our output **still PASSes the identical strict oracle** it passed before. Never trade a bit-exact/structural pass for speed.
2. **Never weaken or delete an oracle** to win (parent §11). Oracle files must be byte-identical pre/post session — assert this in close-out.
3. **A rival doing *less* work is not a bar we lower to.** If a rival is faster only because it emits less than our correctness contract requires yet still trips a weaker effective check, **matching their laziness is legitimate iff our output still passes OUR strict oracle**. If beating them would require producing less than our oracle validates, the cell is **parity-exempt**: record an ADR stating we do strictly-more validated work, and mark it exempt in the generator (with the reason) — this is an honest tie, not a loss, and **not** a license to fake.
4. **Bogus rival metrics don't count.** `peakMemory: 0` from a rival means *unmeasured*, not zero — verify a rival actually instruments a metric before treating its value as a target. Compare only metrics both engines genuinely measure.
5. **Fresh, multi-sample, apples-to-apples.** Re-measure `n≥5` with warmup, on the **same browser (chromium)** the leaderboard scores, same fixtures, same harness build. Never accept the `n=1` export numbers as the final proof (parent §6.3).
6. **Learn technique, don't copy code.** You may read rival source/behaviour (§7) to understand *why* they win; you then **re-implement the idea as our own SOTA TypeScript**. Never copy their code, never vendor it, never delegate the op to a competitor and call it our result (parent §0.6).
